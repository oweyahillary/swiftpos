## 3. ⭐ Priority fix for tomorrow — #1 Expenses "Save Expense" 500

A core feature is broken and the failure is **invisible** (the red "Internal server
error" banner renders below the modal fold with no scroll, so it looks like a dead
button). Two parts: the backend 500 (unknown exact cause) + the off-screen banner (frontend).

**The exact cause is already logged — we just need the line.** `apps/server/src/lib/
sendError.ts` logs every 500 as `[error <ref>] 500 — <real Postgres message>`, and in
dev it also returns a `detail` field in the response body.

### How to get the log line (do this first)
1. Start the API with its console visible: `cd apps/server && npm run dev` (tsx watch).
2. In the dashboard, reproduce: Dashboard → **Expenses → + Add Expense** → fill Category,
   Description, Amount → **Save Expense**.
3. Read the terminal — grab the line that looks like:
   `[error 3f9a1c22] 500 — <message | details | hint | code>`.
   - Alternatively: open DevTools → **Network** → the failed `POST /api/expenses` →
     **Response** tab → copy the `detail` field (present because it's a dev server).
4. Paste that message into the next session.

### What the message will tell us → the matching fix
- `Could not find a relationship between 'expenses' and 'users'` / ambiguous embed →
  the insert's `.select()` embeds `users ( name )` via `paid_by`; if `expenses` has two
  FKs to `users` (e.g. `paid_by` + `created_by`) it's ambiguous. **Fix:** `users!paid_by ( name )`
  (and match the GET list). Same pattern we used for combos.
- `null value in column "…" violates not-null` → the insert omits a required column.
  **Fix:** add it (e.g. `created_by: req.userId`).
- `column "…" does not exist` → schema drift; align the insert to the real columns.

Note: the full `expenses` table definition is **not in this repo's `migrations/`**, so we
can't confirm from source alone — the log line is the shortcut. Handler is
`apps/server/src/routes/expenses.ts` `POST /` (~line 196).

**Frontend part (independent, do alongside):** make the expense modal show its error
banner inside the viewport (scroll or reposition). This is the same silent-failure
pattern as #3/#4 — see the "surface API errors" item in Next steps.
