# SwiftPOS — Engineering Handoff

_Last updated: 2026-07-10_

---

## 1. Goal

Get the multi-tenant Kenyan restaurant POS ("SwiftPOS") audited, hardened, and
stable enough to deploy to the **first paying client**. Work has covered a full
security audit plus ~12 rounds of QA-driven bug fixing across the backend and both
frontends, driven by hands-on testing of the real process flows (table service,
cart, checkout, shifts, reports).

**Stack**
- Backend: Node/Express + TypeScript, hosted on Render
- Data/Auth: Supabase (Postgres, Auth, Realtime, Storage)
- Frontends: React/Vite — `apps/dashboard` (owner + POS) and `apps/admin` (fleet portal)
- Desktop: Electron app — **out of scope** for this work

---

## 2. Current State

**Checkout works.** Confirmed end-to-end by testing: a real sale (African
Breakfast, KES 250, cash) completed and deducted stock (15 → 14). This was the
single biggest blocker and is now resolved.

**Also confirmed working by testing:** split bill, shift management (float in/out +
close-shift reconciliation math), reservations booking, manager-role staff creation
with granular permissions.

**All fixes are cumulative** in a single patch: `swiftpos-patch.zip` (currently 71
files). Apply by copying `apps/` and `migrations/` over the repo root, then rebuild
and restart. `PATCH_NOTES.md` inside the zip is the round-by-round changelog.

**Not yet done by the operator (required before/at go-live):**
- Apply the **latest** patch + rebuild **both** frontends + restart the API
- Run the two DB migrations (see §7)
- Rotate exposed secrets + set required env vars (see §7)
- Verify the two most recent fixes (reporting timezone, table auto-clear)

> ⚠️ **Recurring gotcha — stale builds.** Many "still broken" reports across this
> project turned out to be stale builds. On Windows/Vite, hot-reload misses
> file-overwrites. After applying a patch: copy the **whole** `apps/` tree,
> **restart** each dev server (`npm run dev`), hard-refresh the browser, and
> restart the API. Several bugs reported as "not fixed" were fixed in a patch the
> tester hadn't rebuilt against.

## 4. Changes Made (summary by theme)

**Security audit (all bundled):** forgeable M-Pesa callback → IP allow-list +
validation + idempotency; unauth KDS endpoints → JWT required; hardcoded fallback
secrets → throw if unset; webhook SSRF guard; deactivated-staff token rejection;
JWT alg pinned to HS256; in-memory M-Pesa state → DB; onboarding atomicity; error
leakage → shared `sendError` (generic message + server-logged ref) across ~37
files; timing-safe compare crash; discount tenant check.

**Checkout (the headline fix):** the cart stored each variant as the raw option
`{id, name, price_adjustment}` and discarded the group, but the backend keys on
`{groupName, optionName}` → `"Unknown variant: undefined/undefined"` → 400 on 100%
of variant orders. Fixed the frontend to emit the canonical shape **and** made the
backend accept the option `id` as a fallback. Plus: idempotency key + synchronous
double-submit guard, named missing-field errors, empty-branch guard.

**Shifts:** close-shift 500 fixed via migration 22 (missing `expected_cash` /
`cash_variance` columns + `float_transactions` table) and nullable `notes`.

**Session-expiry (POS + Admin):** the POS client only handled `PERMISSIONS_CHANGED`
401s; now any 401 refreshes+retries then cleanly returns to the PIN lock with a
message. The admin app ignored 401s entirely (→ silent "0 clients"); now clears the
token and returns to login. Admin fleet dashboard no longer renders misleading
zeros on a failed load.

**Reporting / dates:** systemic `toISOString().slice(0,10)` (UTC) → local-date
helper; backend `getDateRange` now interprets filter dates as the EAT business day.

**Stability / UX:** ErrorBoundary now wraps POS/Manager (a receipt render crash no
longer blanks the terminal); ReceiptView null guard; Print Bill rewritten to a
hidden-iframe print (was freezing the tab); Post-to-Room payload rebuilt to the
order API contract; table auto-clears on payment; Z-report spacing; category-card
truncation; staff PIN email; owner password reset feature (admin).

**Pricing/hosting (advice, non-code):** 3 annual per-branch tiers
(Starter 18k / Restaurant 36k / Pro 54k) + 10k onboarding, billed annually upfront
via M-Pesa Paybill; first client as a discounted launch partner; Supabase Pro +
always-on host ≈ $32/mo floor. Editable financial model delivered
(`SwiftPOS_Financial_Model.xlsx`).

## 5. Failed Attempts / Misdiagnoses (so they aren't repeated)

Honest record of what went wrong before the real fix — useful context.

- **Checkout 400:** chased `idempotency_key` (Zod strips unknowns — not it), then
  `branch_id`, before a tester's fetch monkey-patch revealed the true cause: the
  **variant field-name mismatch** in `useCart.ts`. Lesson: the payload capture
  found it in one step; go there first for contract bugs.
- **Shift-close 500:** initially assumed the `closeErr` path; the generic "Internal
  server error" actually meant an uncaught throw, and the real gap was
  missing schema (migration 22). Got the exact mechanism wrong once.
- **Session-expiry:** first fix added a listener for an event the POS client never
  emitted; the real fix was in `posRequest`'s 401 handling.
- **Double-submit:** the first guard stopped the double-charge but exposed a
  `ReceiptView` null-crash with no error boundary (fixed after).
- **Post to Room:** reported "fixed" in round 11, re-reported "broken" in round 12
  — it was a stale build; the fix was already in the bundle.

## 6. What to Test Before Proceeding

Ordered roughly by importance. Do these against a **freshly rebuilt** app.

1. **Variant sale** — add a product with a variant (e.g. size/option), charge cash.
   Should complete (this was the 100%-blocker).
2. **Reporting** — make a sale, then check **Cockpit "Revenue Today"** and the
   **Reports page for today's date**. Both should reflect the sale (this is the fix
   I could not verify without your data).
3. **Date defaults** — open Z-Report, Reports "Today", and the Reservation form;
   each should default to **today**, not yesterday.
4. **Table auto-clear** — open a table, add items, charge, then close the receipt
   **without** using "New Sale" (navigate away). Table should be **free**.
5. **Post to Room** — fill room number, submit; should post a completed sale (no
   "Missing required fields").
6. **Print Bill** — click it; should open a print dialog, **not freeze** the tab.
7. **Transfer Table** — dialog text should read a real table name (or "this order"),
   not "Move 's order…".
8. **Session expiry (both apps)** — let a POS PIN session and an admin session sit
   idle; on next action each should show a clear "session expired" message and
   return to login/lock (not silent zeros / broken screen).
9. **Staff via PIN Login with email** — create one; confirm the email saved and
   that staffer can log into the POS.
10. **Owner password reset (admin)** — client detail → Reset Password → confirm the
    owner can log in with the new password.
11. **Negative-price product** — try to create one; should be rejected.
12. **Shift close** — open a shift, record float, close with a variance note;
    reconciliation should compute correctly (already confirmed once).

## 7. Next Steps / Skipped / Proposed

### Must-do operational (only you can do these)
- **Apply the latest patch, rebuild both frontends, restart the API.**
- **Run migrations** in the Supabase SQL editor: `21_mpesa_payment_tracking.sql`,
  `22_shift_close_and_float.sql`.
- **Rotate exposed secrets** (they were in the original zip): Supabase service-role
  key, `JWT_SECRET`, `ADMIN_JWT_SECRET`, `TECH_HMAC_SECRET`, Resend/SMTP creds.
- **Set env vars:** `ADMIN_JWT_SECRET`, `TECH_HMAC_SECRET` (server won't boot
  without these), `MPESA_ALLOWED_IPS`, `MPESA_ENVIRONMENT=production`,
  `NODE_ENV=production`, `VITE_SUPPORT_WHATSAPP` (real number — rebuild dashboard),
  and optionally `VITE_PRINT_SERVER_URL` (leave unset to disable the print bridge).
- If pushed to GitHub, confirm `.env` was never committed before the repo goes anywhere.

### Known broken / needs a follow-up build
- **KDS (kitchen display)** sends no auth token, so it 401s against the hardened
  kitchen endpoints. Needs to authenticate. Demo on an unpatched backend or fix
  before relying on a kitchen screen.

### In progress / to verify
- **eTIMS (KRA), M-Pesa, On Account** — flagged by tester as still in progress;
  not fully tested. Verify before enabling for a live client.
- **Reporting timezone + table auto-clear** — just shipped; verify per §6.

### Proposed / deferred features (not bugs)
- **Cart persistence** across reload/crash — needs a server-side draft-order model.
- **Unbuilt Menu items:** Hold Orders, Create Orders, mid-sale Customers, Products,
  Payments, Discounts — currently placeholder screens. Hold Orders and mid-sale
  Discounts are ones a restaurant will expect; put on the roadmap.
- **Inventory → POS stock enforcement** — POS currently lets out-of-stock items be
  added (by design for made-to-order food). Add a warning/hard-block if you want it
  enforced — a business decision.
- **Auth consolidation** — move owner login off Supabase into the existing in-house
  auth (staff already use it). Reasonable and bounded (~a few days) because the hard
  parts exist; **do it last**, after the first client is live and the schema is
  frozen, never mid-onboarding. Bank now: give any new `created_by`-style column a
  **nullable** ref and **no hard FK** to `users` (this caused the shift/inventory
  500s). Scope the realtime-cockpit's Supabase-session dependency first.

### Offered, not built
- "Force password change on next login" flag after an owner reset.
- Competitor pricing one-pager (Techlion / SimbaPOS / Cute Profit / EliteTeQ).
- 12-month monthly cash-flow tab for the financial model.
- Consolidated git `.patch` file and a one-step copy-and-restart script.

### Cosmetic / low priority
- "Opening T# — how many guests?" dialog renders light-mode in the dark POS theme.
- `POSDrawer.tsx` React CSS shorthand warning (`border` vs `borderBottomColor`).
- Mismatched product images (African Breakfast, Ugali managu show a book-cover) —
  bad `image_url`s in seed data.
- Category names accept long/duplicate values — length is capped in the schema but
  there's no uniqueness constraint (would need a small migration).

## 8. Key Working Notes

- **Cumulative patch:** `/mnt/user-data/outputs/swiftpos-patch.zip` — apply by
  copying `apps/` + `migrations/` over the repo root. `PATCH_NOTES.md` inside is the
  full changelog.
- **Baseline server `tsc`** has ~130 pre-existing type errors; the build transpiles
  past them. Don't be alarmed by them — check only that *new* changes don't add
  errors and that `npm run build` exits 0 for each frontend.
- **Self-check before assuming a code bug:** grep your local file for the fix string
  first (e.g. `grep -c "onPaid" apps/dashboard/src/pages/pos/PaymentModal.tsx`) — if
  it's there, you're on a stale build.
